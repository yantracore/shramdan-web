# Shramdaan User Journeys

## Version Context

Version 1.0 is the first cleanup campaign's complete visible feature set: from issue listing to publishing the final campaign results.

## Journey Principles

Each journey should feel simple, local, and action-oriented. The user should quickly understand what the problem is, why it matters, how much support it has, and how they can help.

For the first frontend version, journeys may use mock or static data where backend behavior is not ready.

## Issue Reporter

An issue reporter notices a cleanup problem and wants the community to act.

Expected journey:

1. Opens the website.
2. Finds a way to list or understand local issues.
3. Provides issue title, location, description, category, and optional images.
4. Sees the issue appear as a listed community problem.
5. Tracks support and campaign readiness.

Initial frontend behavior may show this as a guided submission concept or prefilled sample issue flow.

## Voter Or Supporter

A supporter wants to help prioritize important issues.

Expected journey:

1. Browses listed issues.
2. Filters or scans by location, status, category, or support count.
3. Opens an issue detail page.
4. Votes, supports, or marks interest.
5. Sees the issue gain visibility or move closer to campaign readiness.

## Volunteer

A volunteer wants to donate labor or time.

Expected journey:

1. Discovers an upcoming cleanup campaign.
2. Reviews time, location, work needed, and expected duration.
3. Chooses how they can help.
4. Confirms participation intent.
5. Receives clear next-step information such as meeting point, supplies, and schedule.

In version 1.0 frontend, confirmation can be simulated without real account or notification infrastructure.

## Donor

A donor wants to contribute funds, materials, or logistics.

Expected journey:

1. Opens a campaign detail page.
2. Reviews what support is still needed.
3. Selects fund, material, or logistics contribution intent.
4. Sees how the contribution would help the campaign.
5. Sees transparency expectations for post-campaign reporting.

Production payments are out of scope for the first frontend version unless added later.

## Organizer

An organizer helps move a supported issue into an executable campaign.

Expected journey:

1. Reviews high-support issues.
2. Confirms that the issue is suitable for community action.
3. Defines campaign date, time, location, required help, and target outcome.
4. Tracks volunteers and contribution needs.
5. Publishes completion summary after the campaign.

The first frontend may represent organizer behavior through campaign status, sample dashboards, or static campaign detail sections.

## Nearby Resident

A nearby resident needs quick local relevance.

Expected journey:

1. Sees issues or campaigns tied to a location.
2. Understands whether the problem affects their area.
3. Supports, joins, shares, or follows the campaign.
4. Sees final proof when the work is completed.

## Public Visitor

A public visitor may not participate immediately but wants to understand the movement.

Expected journey:

1. Lands on the public website.
2. Understands the mission quickly.
3. Sees active issues, upcoming campaigns, and completed results.
4. Reads an impact story.
5. Gains confidence that Shramdaan is practical and transparent.

## How To Update This Document

Update this document when a new user role, journey, or major flow is added. Keep journeys readable and outcome-focused. Detailed page layout belongs in the website structure document.
